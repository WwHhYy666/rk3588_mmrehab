# 动作质量评分及语音交互助手

## 当前接入口径

完成度模型只作为训练后的异步增强能力，不参与摄像头采集、姿态检测、实时计数和训练语音播报。

当前三组动作都使用 `quality_model/models/{action_id}/model.onnx`，后端走 `onnx_cpu`。训练过程中后台 worker 可以提前计算完成度，但 UI 的重点展示放在训练结束后的右侧悬浮栏。

现在展示给患者看的完成度不是纯 ONNX 原始分，而是 `ONNX 原始分 + ROM/TUT 严重度校准`：`raw_quality_score` / `model_score` 保留模型原始输出，`quality_score` / `completion_percent` 用于 UI 和平均值展示。这样 OK 仍保持高分，轻微不到位不会被打得过低，严重高度不足仍会明显低分。

## 完成度显示方式

训练页右侧新增“动作完成度”悬浮页签。

完整训练结束后，这里按固定顺序显示三组动作：

1. 坐站训练 `sit_to_stand`
2. 站姿屈膝后勾腿 `standing_hamstring_curl`
3. 坐姿抬膝 `seated_knee_raise`

每组动作会显示：

- 每一次 attempt 的序号
- 是否计数
- 原始结果：`OK / ROM_LOW / TUT_LOW / 其他错误`
- 中文解释：动作达标、高度不够、保持时间不够等
- 完成度百分比
- 本组平均完成度

平均完成度只统计已经成功推理出完成度的 attempt；如果某次没有分数，该行显示“未评分”，不影响其他 attempt 和训练流程。

## 本地 ONNX 推理验证

已用现有真实 attempt JSON 在本地跑过三组动作，确认模型文件可加载、推理无报错、数值有区分度。

| 动作 | JSON | attempt | 原标签 | 输出完成度 | 后端 | 缓存后推理耗时 |
|---|---|---:|---|---:|---|---:|
| 坐站训练 | `patient_attempt_patient_001_sit_to_stand_20260702_174722.json` | 1 | OK | 96.87% | `onnx_cpu` | 3.05 ms |
| 站姿屈膝后勾腿 | `patient_attempt_patient_001_standing_hamstring_curl_20260702_174908.json` | 1 | OK | 97.39% | `onnx_cpu` | 3.55 ms |
| 坐姿抬膝 | `patient_attempt_patient_001_seated_knee_raise_20260702_182018.json` | 1 | OK | 79.74% | `onnx_cpu` | 6.59 ms |

首次加载模型会比缓存后推理慢。板子上预期应该仍然是后台异步执行，不影响摄像头主流程；请实际上板验证后告诉我结果。

## 20260703 完成度校准回归

已用 2026-07-03 的三份真实 patient attempt JSON 做过回归。结论：

- 坐站训练：OK 仍为 93-98%；轻微 ROM_LOW 约 70.5%；原始模型误给 90.5 的严重 ROM_LOW 被校准到 25.3%。
- 站姿屈膝后勾腿：OK 仍为 96-97%；严重高度不足约目标一半时为 41-42%；TUT_LOW 为 60-67%。
- 坐姿抬膝：入画后的第一个极短 ROM_LOW 假段被过滤，不进入完成度卡片和平均值；真实 OK 仍为 95-98%。

出画后入画会重新进入短暂静止校准，默认约 1.2 秒。刚恢复窗口内，如果出现 `ROM_LOW + 未计数 + duration < 2s + tut_ratio = 0 + rom/rom_target < 0.35`，会被视为入画姿态跳变并静默过滤。

## 麦克风硬件排查

先看系统是否识别录音设备：

```bash
arecord -l
```

正常情况：能看到 USB Audio 或板载 capture device。完全没有设备时，先查 USB 口、麦克风供电、驱动和系统权限。

录 5 秒测试音频：

```bash
arecord -d 5 -f cd test.wav
```

正常情况：命令持续 5 秒后退出，生成非 0 字节的 `test.wav`。

回放测试：

```bash
aplay test.wav
```

正常情况：能听到刚才说话的声音。如果 `arecord` 成功但 `aplay` 没声音，优先查播放设备，不一定是麦克风坏。

如果有多个输入设备，可以按 `arecord -l` 结果指定设备：

```bash
arecord -D plughw:1,0 -d 5 -f cd test.wav
```

### 出画/入画稳定性实测

这轮入画恢复不再把“可以开始动作”当成唯一条件。恢复后会重建动作状态机，重新进入 `reentry_calibrating`，只有连续静止约 1.2 秒、baseline/rest anchor 有效、噪声和速度足够低时，才进入 `reentry_ready` 并允许动作触发计数。

坐姿抬膝在入画后的第一段动作额外启用严格启动门槛：只在 `reentry_strict_start` 窗口内提高启动位移要求，并要求更多连续确认帧，防止刚坐下或轻微晃动被当成 attempt。正常训练窗口外仍使用原来的动作阈值。

建议在 `/status.training.runtime_thresholds` 看这些字段：

```text
reentry_state          是否 reentry_calibrating / reentry_ready
reentry_ready          是否允许入画后重新开始动作
baseline_ready         baseline/rest_anchor 是否已经可信
reentry_sample_count   入画后稳定样本数
rest_noise             静止窗口噪声
velocity               当前动作速度
motion_delta           当前离 baseline 的位移
start_delta_used       本帧实际启动阈值
start_ready            本帧是否达到启动条件
```

板端预期应该是：第二、三组之间出画再入画后，先显示“请保持静止，正在校准”；这时微动不计数，等真正稳定后再做完整动作才计数。请实际验证后把截图或 `/status` 片段发回来。

## 小爱麦克风与唤醒问答

训练页右侧“AI训练图文建议”悬浮栏里有三个入口：

- `麦克风测试`：手动录 2-3 秒，提交 ASR，识别结果会自动填入文字提问框。
- `开启唤醒监听`：只在训练结束、空闲或休息状态使用。
- `提交问答`：把文字框里的问题提交给异步 LLM worker。

### 录音和 ASR 流程

优先使用浏览器端 `getUserMedia` 申请麦克风权限，然后用 `AudioContext` 采集音频，转成 16kHz、16-bit、mono WAV，再以 base64 形式提交：

```text
浏览器录音
-> 16kHz PCM WAV base64
-> POST /api/voice/asr
-> ParaformerAsrWorker 后台线程识别
-> GET /api/voice/asr_result?job_id=...
```

如果浏览器因为 HTTP 非安全上下文、权限或 `navigator.mediaDevices` 不可用而失败，前端会自动切到板端 ALSA：

```text
POST /api/voice/asr_capture
-> 后端调用 arecord 录 2-3 秒 16kHz mono WAV
-> ParaformerAsrWorker 后台线程识别
-> GET /api/voice/asr_result?job_id=...
```

训练进行中 `/api/voice/asr_capture` 会拒绝录音，避免 `arecord` 和 ASR 抢摄像头、计数和播报资源。ASR 结果轮询间隔约 600 ms，最多轮询约 14 秒。

麦克风状态接口：

```text
GET /api/voice/mic_status
GET /api/voice/mic_status?force=1
```

重点看：

```text
mic.arecord_available
mic.capture_devices[]
mic.usb_audio_capture_detected
mic.onboard_rockchip_detected
mic.snd_usb_audio_loaded
mic.summary
```

如果 `arecord -l` 只看到 `rockchip-nau8822`，说明应用层只能尝试板载录音；如果 `snd_usb_audio_loaded=false` 且没有 USB Audio/Jieli/UAC capture，USB 麦克风没有被 ALSA 注册，这属于系统镜像/内核驱动问题，不是前端或录音命令能直接修好的。

如果 UI 显示 `ASR模型缺失` 或 `Paraformer model/tokens not found`，这不是麦克风坏，而是语音识别模型没有放到默认目录。默认目录是：

```text
/home/elf/models/sherpa-onnx-paraformer-zh
```

这个目录里至少要有：

```text
model.onnx 或 model.int8.onnx
tokens.txt
```

也可以通过环境变量改目录：

```bash
export REHAB_ASR_MODEL_DIR=/实际/paraformer模型目录
```

判断顺序是：先看 `/api/voice/status.voice.mic` 是否有录音设备，再看 `/api/voice/status.voice.asr.model_available` 是否为 `true`。录音设备存在但 `model_available=false` 时，麦克风链路可能没问题，缺的是 ASR 模型文件。

## GLM/Qwen UI 一键切换

现在不需要断网才能测试 Qwen。右侧 AI 悬浮栏里有三段按钮：

```text
Auto / GLM / Qwen
```

含义：

- `Auto`：保持默认策略，有 GLM Key 时优先 GLM；本地 Qwen 可用时作为兜底。
- `GLM`：强制走 `glm4v_api`。
- `Qwen`：强制走 `local_qwen_rkllm`，即使有网、有 GLM Key 也走本地 Qwen。

切换接口：

```text
GET  /api/llm/provider
POST /api/llm/provider  { "provider": "auto" | "glm4v_api" | "local_qwen_rkllm" }
```

这个切换只影响当前 8082 进程，重启后会回到启动脚本或环境变量配置。

测试方式：

1. 完成一次训练，确保当前动作有报告。
2. 打开右侧 AI 悬浮栏。
3. 选择 `GLM`，输入“我刚才哪里没做好？”，提交问答。
4. 回答完成后确认显示 `glm4v_api`。
5. 再选择 `Qwen`，提交同样问题。
6. 回答完成后确认显示 `local_qwen_rkllm`。

建议查看字段：

```text
/api/llm/status.llm.provider
/api/llm/status.llm.provider_override
/api/llm/status.llm.active_provider
/api/llm/status.llm.rkllm_server_reachable
/api/llm/status.llm.qwen_generate_ok
/api/llm/status.llm.qwen_generate_cached
/api/llm/status.llm.qwen_generate_age_seconds
/api/llm/status.llm.qwen_generate_error
/api/voice/status.voice.llm_jobs.last_completed_job.active_provider
/api/voice/status.voice.llm_jobs.last_completed_job.qwen_retry_count
/api/voice/status.voice.llm_jobs.last_completed_job.qwen_empty_retry_count
```

Qwen 状态现在分两层：

- `rkllm_server_reachable=true`：只代表 proxy `/health` 或端口可达。
- `qwen_generate_ok=true`：代表最近一次真实 Qwen 问答或 `/generate` smoke test 生成过文本。这个值会缓存约 30 秒，避免 UI 刷新反复抢 NPU。
- `qwen_generate_cached=true`：说明当前显示的是缓存可用性，不是刚刚又跑了一次生成。

如果 UI 显示“Qwen 端口可达 / 生成失败”，优先看：

```text
/api/llm/status.llm.last_error
/api/llm/status.llm.qwen_generate_error
/api/voice/status.voice.llm_jobs.last_completed_job.error
/api/voice/status.voice.llm_jobs.last_completed_job.qwen_retry_count
/api/voice/status.voice.llm_jobs.last_completed_job.qwen_empty_retry_count
logs/qwen_proxy.log
logs/qwen_flask.log
```

`rkllm_proxy_server.py` 会把 upstream 的 `upstream_status`、`upstream_error_preview`、`retry_count` 和 `empty_retry_count` 透出；这能区分“端口通但模型没生成”“返回不是 JSON”“返回 JSON 但没有文本”“RKLLM_Server is busy”等情况。

当前 proxy 会同时向上游发送 `prompt`、`query` 和 OpenAI 风格 `messages`，并兼容 JSON、SSE/stream 行、纯文本、多层 `data/result/output/choices` 等返回格式。它还会把 `/generate` 串行化：同一时刻只让一个请求进入官方 `8080/rkllm_chat`，遇到 `RKLLM_Server is busy` 会短暂等待重试；遇到 `choices[0].message.content` 为空，会自动改用更短 prompt 再试一次。

如果 NPU 占用率变化但仍没有文字，说明请求已经进入本地 RKLLM，问题通常在上游 busy、空文本或 prompt 过长。先看 UI 的最后错误和 `logs/qwen_proxy.log`；如果 proxy 仍连续返回空文本，再单独用官方 `8080/rkllm_chat` curl 测同一句短问题。

预期应该是：选 GLM 时回答走 GLM；选 Qwen 时回答走本地 Qwen。板子上 Qwen 服务是否可用，请以上面字段和 UI 结果为准。
