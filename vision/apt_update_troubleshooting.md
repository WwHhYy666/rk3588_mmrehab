# RK3588 板端 `apt update` 失败排查与替代方案

这份文档专门解决下面这种情况：

- 板子连着网线或 Wi-Fi
- 也能通过 SSH 登上去
- 但是 `sudo apt update` 失败

这种情况通常说明：

- 板子并不是完全离线
- 但也没有真正拿到可用的外网链路

所以最合理的排查顺序不是先换软件源，而是先确认：

```text
本地链路 -> 默认路由 -> 网关 -> 公网 IP -> DNS -> apt 源
```

## 1. 为什么先别急着怀疑 DNS

如果你连下面这种命令都不通：

```bash
ping -c 3 223.5.5.5
```

那你 ping 的已经是纯 IP，不涉及域名解析。

这说明问题更可能在下面这些层面：

- 板子没有默认路由
- 板子只能进局域网，不能出外网
- 网关或 NAT 没有真正放行
- 上游网络策略有限制
- 板子网络配置异常

也就是说：

- 先不要急着改 `apt source`
- 也不要先急着改 DNS

## 2. 第一步：先看网卡和 IP

执行：

```bash
ip addr
```

你要确认：

- 板子有没有拿到有效 IP
- 当前使用的是哪个网卡
- 是 `eth0`、`wlan0`，还是别的接口

如果这里都没有正常 IP，后面的 `apt update` 基本一定不通。

## 3. 第二步：看默认路由

执行：

```bash
ip route
```

重点看有没有类似：

```text
default via 192.168.x.x dev eth0
```

如果没有 `default via ...`，说明：

- 板子知道自己在局域网里
- 但它不知道访问外网应该走哪个网关

这种情况下，`apt update` 失败是很正常的。

## 4. 第三步：先 ping 网关

先从 `ip route` 里找到默认网关地址，再执行：

```bash
ping -c 3 你的网关IP
```

例如：

```bash
ping -c 3 192.168.1.1
```

结果怎么理解：

- 如果网关都 ping 不通：
  - 说明本地链路就有问题
- 如果网关能通，但公网 IP 不通：
  - 更像是 NAT、上游网络或路由放行问题

## 5. 第四步：再 ping 公网 IP

执行：

```bash
ping -c 3 223.5.5.5
ping -c 3 8.8.8.8
```

结果怎么理解：

- 如果两个都不通：
  - 大概率不是 DNS
  - 更可能是板子没有真正出网
- 如果 IP 能通，但域名不通：
  - 才更像是 DNS 问题

## 6. 第五步：再测 DNS

执行：

```bash
cat /etc/resolv.conf
getent hosts mirrors.aliyun.com
getent hosts deb.debian.org
```

如果系统里有 `nslookup`，也可以执行：

```bash
nslookup mirrors.aliyun.com
```

结果怎么理解：

- 如果公网 IP 能通，但域名解析失败：
  - 是 DNS 问题
- 如果公网 IP 本身都不通：
  - 就别先纠结 DNS

## 7. 第六步：最后才看 apt 源

执行：

```bash
cat /etc/apt/sources.list
ls /etc/apt/sources.list.d
```

如果系统比较新，也建议再看：

```bash
grep -R "^[^#].*deb" /etc/apt/sources.list /etc/apt/sources.list.d 2>/dev/null
```

这一步主要是确认：

- 软件源地址是不是写错了
- 是不是指向了一个你当前网络到不了的源
- 是不是源本身已经失效

但前提是：

- 板子本身得先具备正常出网能力

## 8. 按结果分类处理

## 8.1 情况 A：没有默认路由

如果 `ip route` 里没有 `default via ...`，优先处理网络配置。

处理方向：

- 检查 DHCP 是否真的下发了完整配置
- 重新连接网络
- 确认是否拿到了网关
- 必要时临时手动补默认路由

临时验证命令形态：

```bash
sudo ip route add default via 你的网关IP
```

注意：

- 这只是临时验证
- 不是长期配置方案

## 8.2 情况 B：网关能通，但公网 IP 不通

这说明：

- 板子到局域网是通的
- 但上游没有给它真正放行外网

常见原因：

- 路由器自己就没联网
- 当前网络只允许部分设备出网
- 板子接在了只能内网通信的口
- NAT / 防火墙策略有限制
- 电脑共享网络配置不完整

这种情况下，最有效的处理通常不是改 apt，而是换一条更可靠的外网路径先验证。

推荐顺序：

1. 手机热点
2. 一个确认能上网的普通路由器
3. 电脑做网络共享，但要确认 NAT 真生效

## 8.3 情况 C：公网 IP 能通，域名不通

这是典型 DNS 问题。

优先检查：

```bash
cat /etc/resolv.conf
```

如果没有可用 DNS，可以临时测试加公共 DNS，例如：

```text
nameserver 223.5.5.5
nameserver 8.8.8.8
```

但长期建议还是通过系统网络配置把 DNS 正确下发，而不是长期手工修改。

## 8.4 情况 D：IP 和 DNS 都通，但 `apt update` 还是失败

这时再看 apt 源本身。

处理方向：

- 换国内源
- 检查系统版本和源是否匹配
- 看具体报错到底是什么

最常见的几种报错方向：

- `Could not resolve`
- `Connection timed out`
- `404`
- `Release file`

只有拿到完整报错文本，才能继续细分。

## 9. 如果当前环境就是“假联网”，怎么办

如果板子长期都是“能 SSH，但不真出网”，不要把进度卡死在 `sudo apt update` 上。

可以走下面几条替代路线。

## 9.1 方案一：换一个真实外网做一次初始化

最省事的顺序通常是：

1. 手机热点
2. 正常路由器
3. 电脑共享网络

一旦能临时真出网，优先先把下面这些装好：

- `v4l-utils`
- `ffmpeg`
- `python3-pip`
- 当前项目需要的 Python 库

## 9.2 方案二：在电脑下载，再拷到板子

如果板子始终不能联网，可以在电脑上先下载依赖，再传到板子上。

适合对象：

- `.deb` 包
- Python wheel 包
- 源码包

这条路线对下面这些更现实：

- `python-periphery`
- 一些纯 Python 依赖

但对 `mediapipe` 要特别谨慎，因为：

- RK3588 是 `arm64`
- 预编译 wheel 不一定有

## 9.3 方案三：先复用现有环境，绕过 apt

如果你现在板子上已经：

- 能跑 Python
- 能导入 `cv2`
- 能拍照

那当前最优先的就不是把 `apt` 修到完美，而是先继续推进项目验证。

推荐顺序：

1. 先确认现有 Python 环境
2. 先确认 `cv2` 是否可用
3. 再单独解决 `mediapipe` 安装
4. 最后再补系统工具

## 10. 当前项目里最值得先确认什么

如果你的目标只是继续这个 RK3588 视觉项目，那短期最重要的不是“板子是不是一台完美联网机器”，而是先确认：

- Python 环境是否可用
- OpenCV 环境是否可用
- `mediapipe` 能不能装
- 视觉脚本能不能跑

也就是说，当前最实用的目标是：

- 拿到一个“能安装依赖或能运行脚本”的路径

而不是先追求把所有系统联网问题一次性彻底解决。

## 11. 最后的判断标准

你可以按下面这些标准判断自己卡在哪一层：

- `ip addr` 有有效 IP
  - 说明网卡基础连接存在
- `ip route` 有 `default via`
  - 说明系统知道默认外网出口
- `ping 网关` 成功
  - 说明本地链路正常
- `ping 223.5.5.5` 成功
  - 说明公网 IP 通
- `getent hosts deb.debian.org` 成功
  - 说明 DNS 基本正常
- `sudo apt update` 成功
  - 说明软件源链路正常

如果卡住的位置不同，问题层级也不同：

- 卡在网关前：
  - 是本地链路或网卡配置问题
- 卡在公网 IP：
  - 是出网、NAT、上游网络问题
- 卡在域名解析：
  - 是 DNS 问题
- 卡在 apt：
  - 才是源配置问题
