# `vision/` 目录代码说明

这份 `explain.md` 参考了 `hardware/motro_control/explain.md` 的写法，目标是让后来接手的人能看懂：

- `camera_test.py` 是怎么工作的
- `pose_mediapipe_demo.py` 是怎么工作的
- 代码里每一段为什么这样写
- 真正在板子上联调时，优先应该改哪些参数

## 1. 这个目录是干什么的

`vision/` 目录目前是视觉链路的最小验证目录。

它现在有两个核心脚本：

- `camera_test.py`
  - 用来验证 RK3588 板子上的摄像头能不能被 OpenCV 实时打开并显示
- `pose_mediapipe_demo.py`
  - 用来验证 MediaPipe Pose 能不能在板子上对实时视频做人体姿态识别

这两个脚本的定位都不是最终主程序，而是“板端功能验证脚本”。

也就是说，它们现在主要回答的是：

- 摄像头实时预览能不能通
- MediaPipe Pose 骨架识别能不能通

除此之外，当前目录里还有一个辅助脚本：

- `camera_http.py`
  - 用来在 X11 暂时没打通时，通过浏览器快速确认摄像头是否有画面

它的定位不是正式实时预览脚本，而是临时 fallback。

也就是说，当前目录里三份脚本的角色应该这样理解：

- `camera_test.py`
  - 正式的实时预览验证脚本
- `pose_mediapipe_demo.py`
  - 正式的实时骨架预览验证脚本
- `camera_http.py`
  - 临时网页预览脚本

## 2. 代码结构总览

这两个脚本的结构其实很接近，基本都由下面几部分组成：

### 2.1 常量区

这部分把摄像头设备节点、后端类型、图像格式、分辨率等参数集中放在文件开头。

好处是：

- 板端参数集中管理
- 设备节点变了时更容易改
- 不会把这些硬编码散落在主流程里

### 2.2 `open_camera()`

这个函数专门负责：

- 打开摄像头
- 设置图像格式
- 设置分辨率

把这些操作单独封装成函数，有两个直接好处：

- 主流程更清楚
- 如果后续要改摄像头参数，只用集中看这一块

### 2.3 主流程

主流程主要负责：

- 打印启动信息
- 调用 `open_camera()`
- 检查摄像头是否成功打开
- 进入循环读帧
- 显示画面
- 在退出时释放资源

`pose_mediapipe_demo.py` 比 `camera_test.py` 多出来的部分，是：

- 导入 `mediapipe`
- 创建 `Pose` 模型
- 把每帧图像送进模型做识别
- 绘制骨架
- 打印左膝关键点信息

`camera_http.py` 则是另一种完全不同的思路：

- 不使用 `cv2.imshow()`
- 改成在板端起一个 HTTP 服务
- 浏览器通过 MJPEG 图片流去看画面

这就是为什么它虽然也能“看到画面”，但体验和 X11 窗口不一样。

## 3. 常量分别是什么意思

两个脚本开头都有这几个常量：

### 3.1 `CAMERA_DEVICE = "/dev/video21"`

含义：

- 指定当前要打开的摄像头设备节点

为什么重要：

- Linux 板子上的摄像头通常不会像 Windows 那样只写 `0`、`1`
- 这里必须写真实存在的设备节点

当前为什么写 `/dev/video21`：

- 因为你已经确认这个节点能拍照，并且 OpenCV 读图成功

如果以后设备节点变了，最先要改的通常就是它。

### 3.2 `CAMERA_BACKEND = cv2.CAP_V4L2`

含义：

- 指定 OpenCV 打开摄像头时使用 Linux 的 `V4L2` 后端

为什么要显式写出来：

- 这两个脚本已经不再是 Windows 版
- 不能继续用 `cv2.CAP_DSHOW`
- 在 Linux 开发板上，`V4L2` 是更合理、更明确的摄像头后端

### 3.3 `FRAME_WIDTH = 1280`

含义：

- 请求摄像头输出宽度为 1280 像素

### 3.4 `FRAME_HEIGHT = 720`

含义：

- 请求摄像头输出高度为 720 像素

把这两个值合起来，就是：

- 当前请求分辨率为 `1280x720`

为什么现在先用这个值：

- 因为你已经确认这套分辨率可用

如果以后 Pose 太卡，这两个值是最优先考虑调整的参数。

### 3.5 `FOURCC = "MJPG"`

含义：

- 指定摄像头视频流的编码格式为 `MJPG`

为什么要写它：

- 板端摄像头当前已经确认支持 `MJPG`
- 显式指定格式，能减少默认格式不一致带来的问题

## 4. `open_camera()` 这段代码在做什么

两个脚本里都有这个函数：

```python
def open_camera() -> cv2.VideoCapture:
```

它的职责很集中，就是把“打开摄像头并设置参数”这件事封装起来。

函数内部主要做了三件事。

### 4.1 `cv2.VideoCapture(CAMERA_DEVICE, CAMERA_BACKEND)`

作用：

- 按指定设备节点和后端打开摄像头

这里不是直接写 `0`，而是写：

- `CAMERA_DEVICE`
- `CAMERA_BACKEND`

这样更适合板端真实设备。

### 4.2 `cap.set(cv2.CAP_PROP_FOURCC, cv2.VideoWriter_fourcc(*FOURCC))`

作用：

- 告诉摄像头，当前希望使用 `MJPG` 格式输出

为什么不是直接写一个整数：

- `cv2.VideoWriter_fourcc(*FOURCC)` 会把 `"MJPG"` 转成 OpenCV 需要的 FourCC 编码

### 4.3 `cap.set(cv2.CAP_PROP_FRAME_WIDTH, FRAME_WIDTH)` 和 `cap.set(cv2.CAP_PROP_FRAME_HEIGHT, FRAME_HEIGHT)`

作用：

- 请求摄像头输出指定分辨率

注意：

- 这里是“请求”，不是绝对强制
- 真正有没有按这个分辨率输出，还需要结合板端能力和实际测试确认

## 5. `camera_test.py` 主流程解释

这个脚本是最基础的摄像头预览验证脚本。

它的主流程可以理解成下面几步。

### 5.1 打印启动信息

程序开头会打印：

- 程序启动
- 正在打开哪个摄像头设备

这一步的意义是：

- 让你在 SSH 终端里一眼看出程序走到哪一步了

### 5.2 调用 `open_camera()`

这一句：

```python
cap = open_camera()
```

表示：

- 真正去打开摄像头
- 并按当前板端参数配置它

### 5.3 判断 `cap.isOpened()`

这段逻辑是：

```python
if not cap.isOpened():
```

它的作用是：

- 检查摄像头是否真的成功打开

为什么这一步必须有：

- 有时候 `VideoCapture(...)` 调用不直接报错
- 但设备实际上并没有成功打开

如果失败，程序会直接给出中文提示并退出，不会继续往下跑。

### 5.4 循环读帧

核心循环是：

```python
while True:
    success, frame = cap.read()
```

这里的含义是：

- `success` 表示这次读帧是否成功
- `frame` 是当前读到的一帧图像

如果 `success` 是 `False`，说明视频流读取失败，程序就会提示并退出循环。

### 5.5 `cv2.imshow(...)`

作用：

- 把当前帧显示到窗口里

这是最直接的“预览链路验证”。

只要你能稳定看到实时画面，基本就说明：

- 摄像头打开成功
- 视频流能持续读到
- OpenCV 窗口显示也能正常工作

但这里有一个很重要的现实前提：

- 程序运行时必须有可用的图形显示环境

如果你是在板子本地桌面环境里运行，通常问题不大。

如果你是通过纯 SSH 登录板子，再直接运行脚本，就很可能看到类似错误：

```text
qt.qpa.xcb: could not connect to display
Could not load the Qt platform plugin "xcb"
```

这类错误通常不表示：

- 摄像头打不开
- 代码主流程错了

它更常表示：

- `cv2.imshow()` 没有找到可用的显示目标

也就是说，在纯 SSH 场景下，你要么：

- 给板子本地接真正的显示器
- 要么把窗口通过 X11 转发到 Windows
- 要么改走 VNC / 远程桌面

当前这个项目更推荐优先试：

- `X11` 转发到 Windows

### 5.6 `cv2.waitKey(1) & 0xFF`

作用：

- 读取键盘输入
- 给窗口刷新留出时间

为什么要它：

- 没有 `waitKey()`，窗口通常无法正常刷新

### 5.7 退出逻辑

当前写法是：

- 按 `ESC`
- 或按 `q`

程序就退出循环。

这让板端人工验证时更方便。

### 5.8 `cap.release()` 和 `cv2.destroyAllWindows()`

作用：

- 释放摄像头句柄
- 关闭 OpenCV 窗口

这一步很重要，因为如果程序退出时不释放资源，后续再次打开摄像头可能会有占用问题。

## 6. `pose_mediapipe_demo.py` 主流程解释

这个脚本是在 `camera_test.py` 的基础上，多加了 MediaPipe Pose 识别。

## 6.1 `import mediapipe as mp`

作用：

- 导入 MediaPipe Python 包

为什么单独强调它：

- 当前在 RK3588 上最容易出平台兼容问题的就是这里
- 如果 `mediapipe` 装不上，脚本在导入阶段就会失败

## 6.2 `mp_pose = mp.solutions.pose`

作用：

- 拿到 MediaPipe 里的 Pose 方案入口

它相当于告诉程序：

- 这次我们要用的是人体姿态识别方案

## 6.3 `mp_drawing = mp.solutions.drawing_utils`

作用：

- 获取 MediaPipe 自带的绘图工具

后面画骨架时就要用到它。

## 6.4 创建 `Pose(...)` 模型

代码里有这样一段：

```python
pose = mp_pose.Pose(...)
```

这是整个姿态识别的核心。

这里几个参数的含义如下：

- `static_image_mode=False`
  - 表示当前输入是视频流，不是单张静态图片
- `model_complexity=1`
  - 表示使用中等复杂度模型
- `enable_segmentation=False`
  - 表示当前不做人像分割
- `min_detection_confidence=0.5`
  - 表示初次检测人体的最低置信度阈值
- `min_tracking_confidence=0.5`
  - 表示后续跟踪人体关键点的最低置信度阈值

这套参数的特点是：

- 足够适合演示验证
- 不会比更高复杂度模型更吃资源

## 6.5 `cv2.cvtColor(frame, cv2.COLOR_BGR2RGB)`

这一句非常关键。

原因是：

- OpenCV 默认读出来的是 `BGR`
- MediaPipe 需要的是 `RGB`

如果不做这一步转换，识别结果可能异常，甚至完全不稳定。

## 6.6 `result = pose.process(rgb_frame)`

作用：

- 把当前这一帧送进 Pose 模型做识别

它的输出 `result` 里会包含：

- 有没有检测到人体
- 如果检测到了，关键点坐标是什么

## 6.7 `result.pose_landmarks`

作用：

- 表示当前帧的人体姿态关键点结果

如果这个值存在，说明：

- 当前帧识别到了人体姿态

如果这个值为空，说明：

- 当前帧没有识别到可用的人体姿态

## 6.8 `mp_drawing.draw_landmarks(...)`

作用：

- 把识别到的人体骨架画到原始图像上

这样你在窗口里就能直接看到：

- 骨架有没有检测出来
- 关键点连接效果对不对

## 6.9 `landmarks = result.pose_landmarks.landmark`

作用：

- 取出完整的关键点列表

MediaPipe Pose 一般会输出 33 个关键点。

## 6.10 `left_knee = landmarks[25]`

作用：

- 取出左膝关键点

为什么是 `25`：

- 因为在 MediaPipe Pose 的关键点编号定义里，`25` 对应左膝

当前脚本选择它，是因为：

- 左膝对你们骨科康复项目很有业务意义
- 也适合作为关键点打印验证的示例

## 6.11 `visibility`

代码打印的是：

- `x`
- `y`
- `visibility`

其中 `visibility` 的含义是：

- 这个关键点当前在图像里“可见且可信”的程度

一般来说：

- 越接近 `1`，说明越可信
- 越接近 `0`，说明这个关键点越可能被遮挡、丢失或不稳定

## 6.12 为什么每 30 帧才打印一次

代码里用了：

```python
if frame_count % 30 == 0:
```

这么做的目的不是算法要求，而是工程上的可读性考虑。

如果每一帧都打印：

- 终端会刷得非常快
- 很难看清数值有没有变化

每 30 帧打印一次，就能在“看到数据持续更新”和“避免刷屏”之间做一个平衡。

## 6.13 没识别到人体时为什么显示提示文字

脚本里有：

```python
cv2.putText(...)
```

作用：

- 当当前帧没有检测到人体时，在画面上打出提示文字

这样做的好处是：

- 不用只盯终端
- 在窗口画面里也能立刻知道当前状态

## 6.14 退出前为什么还要 `pose.close()`

作用：

- 主动释放 Pose 模型相关资源

这和释放摄像头一样，都属于更完整的资源清理。

## 7. 当前脚本和板子最相关的是哪些地方

真正和 RK3588 板端强相关的，主要就是这些参数：

1. `CAMERA_DEVICE`
2. `CAMERA_BACKEND`
3. `FOURCC`
4. `FRAME_WIDTH`
5. `FRAME_HEIGHT`

把它们理解成：

- 摄像头节点
- 摄像头后端
- 摄像头输出格式
- 摄像头输出分辨率

只要板端真实环境变了，最可能先改的就是这几个地方。

## 8. 如果后面板端环境变化，优先改哪里

### 8.1 如果摄像头节点变了

优先改：

- `CAMERA_DEVICE`

比如 `/dev/video21` 变成了 `/dev/video0`，那就改这里。

### 8.2 如果摄像头格式变了

优先改：

- `FOURCC`

比如设备不再稳定支持 `MJPG`，而是要改成别的格式，就从这里开始改。

### 8.3 如果分辨率变了

优先改：

- `FRAME_WIDTH`
- `FRAME_HEIGHT`

### 8.4 如果帧率不够、程序太卡

优先考虑：

- 先降低分辨率
- 再看是否需要做采集和推理分离

当前阶段最不建议一上来就大改主流程，因为现在还处在验证脚本阶段。

## 9. 这两个脚本当前还不做什么

为了避免后面的人误判，这里也要明确：

当前这两个脚本还没有做下面这些事情：

- 没有接动作评估逻辑
- 没有接 DTW / ROM / TUT
- 没有接数字处方
- 没有接 GPIO 反馈
- 没有接主线程架构

所以它们当前最合适的定位仍然是：

- 板端视觉功能验证脚本

## 10. 后续接手的人最应该记住什么

如果后面你要继续扩展这个目录，最应该先记住这几点：

1. 先保证摄像头预览稳定
2. 再保证 Pose 能稳定识别
3. 板端参数集中看文件开头常量区
4. 节点、格式、分辨率变化时，不要先乱改主流程
5. 如果 `mediapipe` 装不上，优先怀疑平台 wheel 支持问题
